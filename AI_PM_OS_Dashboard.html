    🧠 RCGCF 公式详解

    字母含义示例

    Context背景"我们正在开发一个智能客服系统"
    Goal目标"请编写一份完整的PRD文档"
    Constraint约束"输出不超过2000字，包含用户故事"
    Format格式"使用Markdown格式，分章节呈现"
    RCGCF = Role + Context + Goal + Constraint + Format

    [Context] 我们公司要开发一个 AI 智能客服系统，服务于电商行业的中大型客户。

    🤖 什么是 AI (人工智能)

    AI 定义与分类
    Artificial Intelligence (AI) 是让机器模拟人类智能的技术。

    三大分类：
    • ANI (弱人工智能): 专精单一任务 (Siri, AlphaGo)
    • AGI (通用人工智能): 具备人类级别全面智能 (目标中)
    • ASI (超人工智能): 超越人类所有能力 (未来)
    AI 发展历程
    • 1950年代: 图灵测试、达特茅斯会议
    • 1980年代: 专家系统、神经网络复兴
    • 2010年代: Deep Learning 爆发
    • 2020年代: 大语言模型 (LLM) 时代
    PM 为什么需要懂 AI？
    • 判断 AI 技术可行性边界
    • 与技术团队有效沟通
    • 设计合理的 AI 产品功能
    • 预判开发成本和时间
    • 理解用户对 AI 的期望

    🎯 Machine Learning (机器学习)

    ML 核心概念
    Machine Learning = 用数据训练模型，让机器自动学习规律

    三大范式：
    1. 监督学习: 有标签数据 (分类、回归)
    2. 无监督学习: 无标签数据 (聚类、降维)
    3. 强化学习: 奖励机制 (游戏 AI、机器人)
    常见 ML 算法
    • 线性/逻辑回归 - 基础预测模型
    • 决策树/随机森林 - 可解释性强
    • SVM - 分类问题经典算法
    • K-Means - 聚类分析
    • 神经网络 - Deep Learning 基础
    PM 应用场景
    • 推荐系统 (协同过滤 + 内容推荐)
    • 用户分群 (聚类算法)
    • 风控模型 (异常检测)
    • 定价优化 (回归预测)

    🧪 Deep Learning (深度学习)

    什么是 Deep Learning？
    Deep Learning = 多层神经网络的机器学习方法

    核心优势：
    • 自动特征提取 (无需人工设计特征)
    • 处理非结构化数据 (图像、文本、语音)
    • 在大数据量下表现卓越
    • 端到端学习 (输入原始数据→输出结果)
    关键架构演进
    • CNN (卷积神经网络) - 图像识别 (AlexNet, ResNet)
    • RNN/LSTM - 序列数据处理
    • Transformer - Attention 机制革命
    • GAN - 生成对抗网络 (图像生成)
    • Diffusion Model - 扩散模型 (Midjourney, Stable Diffusion)
    DL vs 传统 ML 对比
    传统 ML: 数据量小可工作、可解释性好、需要特征工程
    Deep Learning: 需要大量数据、黑盒难解释、自动学特征

    💡 NLP (自然语言处理)

    NLP 定义
    Natural Language Processing = 让计算机理解、生成人类语言

    核心任务：
    • 文本分类 (情感分析、垃圾邮件检测)
    • 命名实体识别 (NER) - 提取人名、地名、机构
    • 机器翻译 (Google Translate)
    • 问答系统 (ChatGPT)
    • 文本摘要 (自动生成摘要)
    NLP 技术演进
    • 规则方法 - 人工编写语法规则
    • 统计方法 - HMM, CRF
    • Word Embedding - Word2Vec, GloVe
    • 预训练模型 - BERT, GPT 系列
    • 大语言模型 - GPT-4, Claude, Gemini
    Tokenization (分词)
    Token = LLM 处理文本的最小单位

    • Word-level: 按单词切分 ("Hello" → ["Hello"])
    • Subword-level: BPE, WordPiece (处理未知词)
    • Character-level: 按字符切分

    为什么重要？影响模型输入长度限制和成本计算

    ⚙️ 关键参数详解

    Temperature (温度)
    控制输出随机性的核心参数 (0-2)

    • 低 Temperature (0-0.3): 输出确定、一致、事实性高
      适用：代码生成、数据提取、翻译

    • 中 Temperature (0.5-0.8): 平衡创造性和准确性
      适用：日常对话、写作辅助、问答

    • 高 Temperature (0.9-1.5): 输出多样、创意性强
      适用：创意写作、头脑风暴、角色扮演

    PM 提示：产品中通常设为固定值或让用户选择
    Top-P (核采样)
    另一种控制随机性的方法 (0-1)

    只从累积概率达到 P 的 token 中采样：
    • Top-P = 0.1: 只选最可能的 10% token
    • Top-P = 0.9: 选概率累积到 90% 的 token

    vs Temperature 区别：
    • Temperature 是全局调整
    • Top-P 是动态调整（根据实际概率分布）

    最佳实践：两者配合使用，或只用一个
    其他重要参数
    • Max Tokens - 最大输出长度限制
    • Stop Sequences - 停止生成的触发词
    • Frequency Penalty - 减少重复词出现
    • Presence Penalty - 鼓励新话题引入
    • Top-K - 只从 K 个最可能 token 中选择

    调优建议：先用默认值，根据输出质量微调

    📚 LLM (大语言模型)

    什么是 LLM？
    Large Language Model = 参数量巨大的预训练语言模型

    "大"的定义：
    • 参数量: 通常 > 1B (10亿) 参数
    • 训练数据: TB 级别的文本语料
    • 训练成本: 数百万到数千万美元

    代表模型：GPT-4 (~1.8T), Claude 3, Gemini Ultra, LLaMA 3
    LLM 核心能力
    ✅ 文本生成 - 写作、创作、代码生成
    ✅ 理解推理 - 问答、分析、数学
    ✅ 多轮对话 - 上下文记忆、角色扮演
    ✅ Few-shot Learning - 少样本快速适应
    ✅ Chain-of-Thought - 分步推理解决问题
    LLM 局限性
    ⚠️ 幻觉 (Hallucination) - 编造不存在的信息
    ⚠️ 上下文窗口限制 - 无法处理超长文档
    ⚠️ 实时性差 - 训练截止日期后的事件不知道
    ⚠️ 成本高 - API 调用费用昂贵
    ⚠️ 偏见问题 - 训练数据中的社会偏见
    📏 Context Window (上下文窗口) 详解
    Context Window = LLM 一次能处理的最多 Token 数量

    主流模型上下文长度对比：
    • GPT-4o: **128K** tokens (~300页文本)
    • Claude 3.5: **200K** tokens (~500页文本)
    • Gemini 1.5 Pro: **1M-2M** tokens (~数百万页)
    • LLaMA 3: **8K-128K** tokens

    为什么重要？
    ✅ 决定能否一次性处理长文档
    ✅ 影响多轮对话的记忆长度
    ✅ 直接影响 API 成本 (Token 越多越贵)

    PM 决策：需要处理多长文档？选择合适上下文长度的模型

    🔄 Transformer 架构

    Transformer 革命 (2017)
    论文: "Attention Is All You Need" (Google)

    核心创新 - Self-Attention (自注意力机制):
    让模型在处理每个词时，都能"关注"到句子中的其他相关词汇

    类比：读一句话时，你自然会关注关键词来理解整体含义
    Encoder-Decoder 架构
    Encoder (编码器):
    • 理解输入文本的语义
    • 双向注意力 (同时看前后文)

    Decoder (解码器):
    • 逐词生成输出
    • 单向注意力 (只能看前面的词)

    变体：BERT (纯Encoder), GPT (纯Decoder), T5 (完整架构)
    为什么 Transformer 如此重要？
    • 并行计算 - 比 RNN 快数百倍
    • 长距离依赖 - 解决梯度消失问题
    • 可扩展性 - 更大模型效果更好
    • 通用性 - NLP/CV/多模态统一架构
    • 诞生了所有现代 LLM

    ⚡ Attention Mechanism (注意力机制)

    Attention 工作原理
    QKV 三元组：
    • Query (Q): "我在找什么信息？"
    • Key (K): "我有什么信息可以提供？"
    • Value (V): "实际的内容是什么？"

    计算过程：
    Attention(Q,K,V) = softmax(Q·K^T / √d_k) · V

    简单说：计算 Query 和所有 Key 的相似度，加权求和 Value
    Multi-Head Attention
    多头注意力 = 并行运行多个 Attention 机制

    • 每个 Head 学习不同的关注模式
    • 有的关注语法关系
    • 有的关注语义相似
    • 有的指代消解

    类比：多人从不同角度阅读同一篇文章，综合得出更全面的理解
    Positional Encoding (位置编码)
    问题：Transformer 本身没有顺序概念！

    解决方案：给每个位置添加位置信息
    • Sinusoidal - 正弦余弦函数 (原始方案)
    • Learned - 可学习的嵌入向量 (GPT-2+)
    • RoPE - 旋转位置编码 (LLaMA 使用)

    🎓 Pre-training & Fine-tuning

    两阶段训练范式
    Stage 1: Pre-training (预训练)
    • 在海量无标注数据上训练
    • 学习通用的语言理解和生成能力
    • 成本极高 (数百万美元)
    • 产出: Base Model (基础模型)

    Stage 2: Fine-tuning (微调)
    • 在特定任务的小数据集上调整
    • 让模型适配特定场景
    • 成本相对较低
    • 产出: 专用模型
    Fine-tuning 方法对比
    • Full Fine-tuning - 更新所有参数 (效果好但贵)
    • LoRA (Low-Rank Adaptation) - 只训练低秩矩阵 (成本低70%)
    • QLoRA - LoRA + 量化 (4-bit 训练)
    • RLHF - 人类反馈强化学习 (ChatGPT 核心)
    • DPO - 直接偏好优化 (比 RLHF 简单)
    PEFT (参数高效微调)
    为什么需要 PEFT？
    • 全量微调 7B 模型需要 28GB 显存
    • LoRA 只需 4-8GB 显存
    • 效果接近全量微调
    • 支持快速切换不同任务

    工具库：PEFT (Hugging Face), Axolotl, Unsloth
    🎯 SFT / RLHF / DPO 三大对齐技术
    1. SFT (Supervised Fine-tuning) 监督微调
    • 使用高质量问答对训练模型
    • 让模型学会"如何正确回答"
    • 数据来源：人工标注、GPT-4生成

    2. RLHF (Reinforcement Learning from Human Feedback)
    • ChatGPT 的核心技术！
    • 三阶段：SFT → 奖励模型(RM) → PPO强化学习
    • 人类对回答排序，训练奖励模型
    • 用PPO算法优化策略，最大化奖励
    • 优点：效果最好   缺点：复杂、昂贵

    3. DPO (Direct Preference Optimization)
    • 直接偏好优化，无需训练奖励模型
    • 从偏好数据直接学习最优策略
    • 优点：简单、稳定、成本低   缺点：略逊RLHF

    PM 视角：ChatGPT = Pre-training + SFT + RLHF

    🔗 RAG (检索增强生成)

    什么是 RAG？
    RAG = Retrieval-Augmented Generation
    结合了检索和生成的混合 AI 架构

    核心原理：
    1. 用户提问 → 2. 检索相关知识文档 → 3. 将文档作为上下文 → 4. LLM 生成回答并引用来源

    解决的核心问题：LLM 幻觉、知识过时、私有数据访问
    RAG vs Fine-tuning 选择指南
    选择 RAG 当你需要：
    ✅ 动态更新的知识库
    ✅ 可追溯的回答来源
    ✅ 降低幻觉风险
    ✅ 利用企业私有数据

    选择 Fine-tuning 当你需要：
    ✅ 特定风格或格式输出
    ✅ 领域专业术语理解
    ✅ 低延迟响应
    ✅ 减少 Token 消耗
    RAG 进阶技术
    • Hybrid Search - 关键词 + 向量混合检索
    • Reranking - 二次排序提升精度 (Cohere Rerank)
    • Parent-Child Chunking - 父子文档策略
    • Multi-query - 一个问题生成多个查询变体
    • Agentic RAG - Agent 自主决定检索策略

    🗄️ Vector Database (向量数据库)

    为什么需要向量数据库？
    传统数据库擅长精确匹配，但无法做"语义搜索"

    例子：
    • 传统 SQL: WHERE title = 'AI产品' → 只能找到完全匹配
    • 向量搜索: 找到语义相近的 "机器学习应用"、"人工智能工具"...

    核心技术：将文本转为高维向量 (768/1536维)，计算余弦相似度
    主流向量数据库对比
    • Pinecone - 全托管云服务，最易上手
    • Weaviate - 开源 + 云服务，支持混合搜索
    • Milvus - 开源，性能强，适合大规模
    • Chroma - 轻量级嵌入式，适合本地开发
    • Qdrant - Rust 写成，高性能过滤
    • pgvector - PostgreSQL 扩展，无需额外数据库
    PM 选型决策树
    选云服务如果：团队小、不想运维、预算充足 → Pinecone
    选开源自建如果：数据敏感、需要定制、有运维能力 → Milvus/Qdrant
    选轻量级如果：原型验证、个人项目 → Chroma
    选 PostgreSQL 如果：已有 PG 实例、需要事务一致性 → pgvector

    📐 Embedding Model (嵌入模型)

    Embedding 原理
    Embedding = 将文本映射为高维空间中的向量

    关键特性：
    • 语义相似的文本 → 向量距离近
    • 语义不同的文本 → 向量距离远
    • 维度通常: 384 / 768 / 1536 维

    常用距离度量：
    • 余弦相似度 (Cosine Similarity) - 最常用
    • 欧氏距离 (Euclidean Distance)
    • 点积 (Dot Product)
    主流 Embedding 模型
    • OpenAI text-embedding-3 - 小/大尺寸可选 (1536维)
    • Cohere embed-v3 - 多语言支持好
    • HuggingFace E5 - 开源 SOTA
    • Voyage AI - 检索专用优化
    • BGE - 中文优化 (智源研究院)
    • M3E - 中文开源推荐
    Embedding 成本估算
    以 OpenAI 为例：
    • text-embedding-3-small: $0.02 / 1M tokens
    • text-embedding-3-large: $0.13 / 1M tokens

    100万中文 ≈ 150万 tokens ≈ $3-$20

    省钱技巧：用开源模型本地部署 (BGE/M3E)

    📊 RAG 评估框架

    RAGAS 框架
    RAGAS = RAG Assessment - 专门评估 RAG 系统质量

    四大评估维度：
    1. Faithfulness (忠实度) - 回答是否基于检索内容
    2. Answer Relevancy (相关性) - 回答是否针对问题
    3. Context Precision (上下文精确度) - 检索内容是否有用
    4. Context Recall (上下文召回率) - 是否找到所有相关信息
    其他评估指标
    • MRR (Mean Reciprocal Rank) - 排序质量
    • NDCG - 归一化折损累积增益
    • Hit Rate - 命中率
    • Latency - 响应延迟 (P50/P95/P99)
    • Cost per Query - 单次查询成本
    RAG 优化 Checklist
    ☐ Chunk size 调优 (512/1024/2048 tokens 测试)
    ☐ 重叠比例设置 (10%-20% 防止信息断裂)
    ☐ Top-K 值调整 (通常 5-20)
    ☐ Reranker 引入 (提升精度 15-30%)
    ☐ 元数据过滤 (时间范围、文档类型)
    ☐ 缓存策略 (相同问题不重复检索)

    🤖 AI Agent (智能体)

    Agent 定义
    Agent = 具备自主决策和行动能力的 AI 系统

    不只是"回答问题"，而是：
    1. 感知 (Perception) - 接收环境信息
    2. 推理 (Reasoning) - 规划执行步骤
    3. 行动 (Action) - 调用工具/API
    4. 反思 (Reflection) - 根据结果调整策略

    核心区别：LLM 被动回答 vs Agent 主动完成任务
    Agent 架构模式
    • ReAct (Reasoning + Acting)
    交替进行思考和行动，最常用模式

    • Plan-and-Execute
    先制定完整计划，再逐步执行

    • Multi-Agent System
    多个专业 Agent 协作 (CEO/CTO/CMO 角色)

    • Hierarchical Agent
    上层规划，下层执行的层级结构
    Agent 能力边界
    当前能做到：
    ✅ 自动浏览网页、搜索信息
    ✅ 调用 API 完成操作
    ✅ 读写文件、操作数据库
    ✅ 发送邮件、消息通知
    ✅ 编写和执行代码

    当前局限：
    ❌ 长期记忆和持续学习
    ❌ 复杂多步骤任务的可靠性
    ❌ 真正的理解而非模式匹配

    🛠️ Tool Use (工具调用)

    Function Calling 机制
    Tool Use = LLM 决定何时以及如何调用外部函数

    工作流程：
    1. 用户提问 → 2. LLM 分析需要什么工具 → 3. 输出函数调用请求 → 4. 系统执行函数 → 5. 结果返回 LLM → 6. LLM 生成最终回答

    标准协议：OpenAI Function Calling, Anthropic Tool Use
    常用 Tool 类型
    • Search Tools - Google Search, Tavily, SerpAPI
    • Code Tools - Python REPL, Code Interpreter
    • Data Tools - SQL Database, CSV/Excel
    • API Tools - Weather, Stock, Maps
    • File Tools - Read/Write/Create/Delete
    • Communication - Email, Slack, Notion
    Tool 设计最佳实践
    ✅ 清晰的工具名称和描述 (LLM 靠描述选择工具)
    ✅ 明确的参数定义和类型约束
    ✅ 错误处理和重试机制
    ✅ 权限控制和安全沙箱
    ✅ 工具结果的简洁返回 (节省 Token)
    ❌ 不要让 Agent 有破坏性操作的权限

    🔌 MCP (Model Context Protocol)

    MCP 是什么？
    MCP = Model Context Protocol
    Anthropic 提出的开放标准 (2024年11月)

    核心理念：让 AI 模型能安全地与外部数据源和工具交互

    类比：
    • USB 对电脑 - 统一硬件接口
    • App Store 对手机 - 统一软件分发
    • MCP 对 AI - 统一工具集成标准
    MCP 三层架构
    1. MCP Host (宿主)
    运行环境：Claude Desktop, IDE (VS Code/Cursor), Web App

    2. MCP Client (客户端)
    协议实现：负责与 Server 通信

    3. MCP Server (服务器)
    数据源/工具：GitHub, Slack, Google Drive, PostgreSQL, File System...
    为什么 PM 要关注 MCP？
    🎯 降低集成成本 - 一次开发，所有兼容平台可用
    🎯 生态效应 - 类似 iOS App Store 的网络效应
    🎯 标准化机会 - 可能成为 AI 时代的 API 标准
    🎯 竞争格局 - OpenAI 可能推出类似标准 (Plugin v2?)
    🎯 创业机会 - 开发垂直领域 MCP Server

    🏗️ Memory (记忆系统)

    Agent 记忆层次
    1. Short-term Memory (短期记忆)
    • 当前对话上下文 (Context Window)
    • 有限容量 (8K-128K tokens)
    • 会话结束即丢失

    2. Long-term Memory (长期记忆)
    • 跨会话持久化存储
    • 向量数据库存储历史交互
    • 检索时按相关性召回

    3. Episodic Memory (情景记忆)
    • 记住特定事件和经历
    • 时间线组织
    • 用于个性化
    Memory 实现方案
    • Zep - 专为 LLM Agent 设计的记忆层
    • Mem0 - 开源记忆管理框架
    • Letta (MemGPT) - 操作系统风格的记忆管理
    • 自定义方案 - PostgreSQL + pgvector

    关键功能：记忆总结、重要性评分、遗忘机制
    Memory 设计考量
    ⚠️ 隐私合规 - GDPR 要求被遗忘权
    ⚠️ 成本控制 - 存储和检索都有成本
    ⚠️ 准确性 - 错误记忆比没有记忆更糟糕
    ⚠️ 时效性 - 过时的记忆应该降权或删除

    🎯 Skill (技能/能力)

    什么是 Skill？
    Skill = Agent 或 LLM 可学习、可调用的特定能力或专业技能

    核心概念：
    • Skill 是**可复用的能力单元**
    • 类似于人类的"技能"：会写代码、会搜索、会翻译...
    • Agent 可以**动态组合多个 Skill** 完成复杂任务
    • Skill 可以**通过示例学习**或**预定义规则**获得

    类比：就像手机App，每个App是一个Skill，组合使用实现复杂功能
    Skill 类型分类
    1. 基础 Skills (Foundation Skills)
    • 文本生成、总结、翻译
    • 代码编写、调试、解释
    • 数学计算、逻辑推理

    2. 工具 Skills (Tool Skills)
    • 搜索引擎调用、API 请求
    • 数据库查询、文件操作
    • 邮件发送、消息通知

    3. 领域 Skills (Domain Skills)
    • 法律文档分析、医疗诊断辅助
    • 金融数据处理、营销文案撰写
    • 产品设计建议、用户研究分析
    Skill 实现方式

    2. Function-based Skills (基于函数)
    封装为可调用的函数/工具
    例：search_web(query), calculate_formula(expr)

    3. Fine-tuned Skills (微调技能)
    针对特定任务微调的专用模型
    例：法律LLM、医疗LLM、代码LLM

    4. MCP Skills (协议化技能)
    通过MCP标准封装的标准化技能
    例：GitHub MCP Server, Slack MCP Server
    PM 如何设计 Skill 系统？
    ✅ **识别核心场景** - 用户需要什么能力？
    ✅ **定义 Skill 边界** - 单一职责，粒度适中
    ✅ **设计 Skill 组合** - 允许灵活编排
    ✅ **建立 Skill 市场** - 让第三方贡献 Skills
    ✅ **监控 Skill 质量** - 成功率、延迟、满意度

    成功案例：OpenAI GPTs、Claude Projects、Custom GPTs

    🔄 Agentic Patterns (智能体模式)

    核心 Agentic 模式
    1. ReAct (Reasoning + Acting)
    最常用的Agent模式：思考→行动→观察→再思考
    适用：需要推理的多步骤任务

    2. Plan-and-Execute (规划-执行)
    先制定完整计划，再逐步执行和调整
    适用：复杂项目、长期目标

    3. Reflection (反思模式)
    执行后回顾结果，自我纠正错误
    适用：写作、编程、数据分析

    4. Multi-Agent Collaboration (多智能体协作)
    多个专业Agent分工合作（如：研究员+写手+审稿人）
    适用：内容创作、研究报告
    Agentic Workflow 设计原则
    ✅ **明确目标** - Agent知道要完成什么
    ✅ **工具就绪** - 提供必要的Skills和Tools
    ✅ **反馈循环** - 能根据结果调整策略
    ✅ **容错机制** - 失败时有重试或备选方案
    ✅ **人类监督** - 关键决策点有人工确认

    框架支持：LangGraph, CrewAI, AutoGen

    📐 Structured Output (结构化输出)

    为什么需要结构化输出？
    问题：LLM 默认输出自由文本，难以程序化处理

    解决方案：强制输出特定格式
    • JSON 格式 - 最常用，易于解析
    • XML 格式 - 支持嵌套结构
    • TypeScript 类型 - 前端友好
    • 自定义 Schema - 按需定义

    应用场景：API响应、数据库写入、表单填充
    主流实现方案
    • Structured Outputs (新) - 定义JSON Schema，100%符合格式
    • Anthropic Tool Use - 通过function calling输出结构化数据
    • Pydantic / Zod - 定义数据模型，验证输出格式
    • Outlines / Guidance - 约束解码，强制格式合规

    最佳实践：定义清晰的Schema + 提供Few-shot示例
    Function Calling 进阶
    基础用法：单次调用一个函数

    进阶模式：
    • Parallel Calling - 同时调用多个函数 (GPT-4o支持)
    • Nested Calling - 函数结果作为另一个函数的输入
    • Conditional Calling - 根据条件选择不同函数
    • Streaming Function Calls - 流式输出函数调用结果

    PM 设计要点：定义清晰的函数接口、处理并发调用、优雅的错误降级

    📊 LLM 评估方法

    评估维度
    1. 质量 (Quality)
    • 正确性 (Accuracy) - 回答是否正确
    • 相关性 (Relevance) - 是否针对问题
    • 完整性 (Completeness) - 信息是否充分

    2. 安全性 (Safety)
    • 有害内容检测
    • PII 泄露检查
    • Bias 偏见评估

    3. 性能 (Performance)
    • Latency (P50/P99 延迟)
    • Throughput (TPS 吞吐量)
    • Cost per Request
    评估方法论
    • 人工评估 (Human Eval)
    最准确但昂贵，适合金标准建立

    • LLM-as-Judge
    用 GPT-4/Claude 评判其他模型的输出
    成本低，与人工评估相关性 ~80%

    • 众包评估 (Scale/AWS MTurk)
    平衡成本和质量

    由 **EleutherAI** 开发的开源评估工具，用于对 LLM 进行全面、标准化的性能测试。

    核心功能：

    ✅ 多模型支持 - HuggingFace、OpenAI、本地模型均可
    ✅ 可扩展 - 自定义评测任务和指标

    知名评测榜单使用：

    🎯 HEMERS 评估框架

    HEMERS 六维评估体系
    HEMERS = Holistic Evaluation Metric for Enterprise Response Systems

    面向企业级 AI 应用的综合评估框架：

    H - Helpfulness (有用性)
    回答是否真正帮助用户解决问题？

    E - Exactness (准确性)
    事实陈述是否正确无误？

    M - Measurability (可度量性)
    评估指标是否客观可量化？
    HEMERS 续
    E - Efficiency (效率性)
    响应速度、资源消耗、成本效益

    R - Robustness (鲁棒性)
    面对异常输入、攻击、边缘情况的稳定性

    S - Safety (安全性)
    隐私保护、内容安全、合规要求

    适用场景：客服机器人、企业助手、金融咨询等生产系统
    HEMERS 实施建议
    ✅ 建立各维度的基线和阈值
    ✅ 定期自动化回归测试
    ✅ 结合业务指标 (CSAT/NPS)
    ✅ 灰度发布时重点监控 R 和 S
    ✅ 建立人工审核抽样机制 (10% 抽检)

    🔒 AI 安全与治理

    主要安全风险
    1. Prompt Injection (提示注入)
    恶意用户通过精心构造的输入绕过安全限制

    2. Data Leakage (数据泄露)
    模型在回复中泄露训练数据或用户隐私

    3. Jailbreak (越狱)
    诱导模型违反安全准则输出有害内容

    4. Hallucination (幻觉)
    生成虚假但看似可信的信息
    安全防护措施
    • Input Guardrails - 输入过滤和检测 (NeMo Guardrails)
    • Output Filtering - 敏感词过滤、PII 识别
    • Content Moderation - 内容审核 API (OpenAI Moderation)
    • Rate Limiting - 防止滥用和 DDoS
    • Audit Logging - 完整的操作日志记录
    • RBAC - 基于角色的权限控制
    合规框架
    • EU AI Act - 欧盟 AI 法规 (分级监管)
    • GDPR - 数据隐私保护 (被遗忘权)
    • AI Ethics Guidelines - 可解释性、公平性、透明度
    • Industry Standards - ISO/IEC 42001 AI管理体系

    📈 AI Product Metrics (AI产品指标)

    AI 产品特有指标
    不同于传统产品的指标体系：

    模型层面：
    • Accuracy/Precision/Recall/F1
    • Perplexity (困惑度) - 语言模型专用
    • BLEU/ROUGE - 生成质量
    • Latency P50/P99 - 响应时间分布

    用户体验层面：
    • Task Success Rate - 任务完成率
    • Time to Complete - 任务耗时
    • User Satisfaction (CSAT/NPS)
    • Retention Rate - 留存率
    成本指标
    💵 Token Consumption - 总消耗 Token 数
    💵 Cost per User - 单用户成本
    💵 Cost per Query - 单次查询成本
    💵 Cache Hit Rate - 缓存命中率 (直接影响成本)
    💵 Model Utilization - 模型利用率

    目标：在保证质量的前提下最小化成本
    A/B Testing for AI
    AI 产品 A/B 测试要点：
    ✅ 测试变量：不同模型、不同 Prompt、不同参数
    ✅ 分流策略：确保用户群体一致
    ✅ 样本量：AI 交互方差大，需要更大样本
    ✅ 统计显著性：注意 p-value 和置信区间
    ✅ 长期跟踪：短期效果 ≠ 长期留存

    工具：LaunchDarkly, Optimizely, Statsig
